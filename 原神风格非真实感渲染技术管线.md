3. **[合理推断]** 社区对角色资源的分析普遍得到“控制贴图 + 多行 Ramp + 面部阈值图 + 反面外扩描边”这一可复现结构，[C2][C3][C4]；HoyoToon 当前代码也实现了相同模块组合。[C1] 但各角色、版本、皮肤的通道语义并不恒定，不能把一张角色贴图的通道表推广为全游戏规范。

6. **[已证实]** 附件标准主体 `桑多涅.pmx` 是 PMX 2.0 的 MMD 改造模型：61,973 顶点、69,864 三角形、31 个材质、692 根骨骼、64 个 Morph、18 个纹理引用。

8. **[通用方案]** 米哈游 2018 年《崩坏3》公开演讲展示了多通道 Ramp、面部漫反射抑制、切线各向异性头发、顶点色控制的透视校正几何描边，以及深度/法线屏幕描边。[O2] 这些可证明米哈游系实时动漫渲染的技术谱系，但不能证明《原神》逐项沿用了相同实现。

| 模块 | 主要输入 | 主要输出 | 关键依赖 | 失败时的可见症状 |

| 贴图准备 | Base、遮罩、AO、材质 ID | BaseMap、ControlMap、Ramp、FaceMap | 线性/色彩数据区分 | 通道阈值漂移、颜色发灰 |
| 主光与阴影 | 光方向/颜色、Shadow Map | `Light`、阴影衰减 | ShadowCaster、级联设置 | 漂浮、Peter Panning、漏影 |
| 身体 Toon | $N,L,V$、控制贴图、Ramp | 漫反射/高光 | 主光、材质 ID | 塑料感、色阶抖动 |
| 面部 | 头部前/右轴、光方向、Face SDF | 面部明暗遮罩 | 稳定头骨坐标、对称 UV | 阴影反向、随身体旋转错误 |

- **[已证实]** `tex/颜.png`、`体.png`、`髮.png`、`裙.png`、`肌.png` 是带大量手绘明暗与轮廓细节的 Base 纹理；应以 sRGB 颜色输入导入。
- **[已证实]** `hair.bmp`、`skin.bmp` 为 32×32，`toon_defo.bmp` 为 16×16；它们是 MMD 小型 toon 查找纹理，不是社区复刻中“多材质行 × 昼夜”的 ILM/Ramp 契约。
- **[已证实]** `spa/hair_s.bmp`、`spa/stk.png`、`spa/目sp.png` 等是 MMD sphere-map 风格输入；其方向/混合语义与 Unity Lit 的 Metallic/Smoothness 不等价。

### 3.3 为风格化渲染设计几何

**[通用方案]** 二次元角色模型需要优先保证投影视图，而不是均匀追求物理体积：

- 面部正面和 3/4 视图要作为拓扑验收视角；鼻梁、嘴唇不宜产生过强真实阴影。
- 头发大块面应形成清晰节奏；刘海与脸之间保留足够深度，避免 Z-fighting 和错误接触阴影。
- 轮廓敏感区域增加均匀顶点密度。反面外扩描边是逐顶点变形，过长三角形会造成轮廓折线。
- 眼睛通常不是单一透明平面：眼白、虹膜、瞳孔、高光和睫毛需要明确的遮挡关系。
- 为头部保存稳定的 `HeadForward`、`HeadRight`、`HeadUp` 基准；面部阈值不能依赖世界坐标固定轴。

## 4. 法线、顶点数据与贴图体系

### 4.1 三种法线不能混为一谈

1. **几何/着色法线**：控制漫反射和高光。可为脸和身体做艺术化编辑，使阴影更接近平面插画。
2. **切线空间法线贴图**：增加细节，不改变轮廓。Unity 以 Y+（OpenGL 约定）为默认；若源贴图相反，可用 `Flip Green Channel`。[U5]
3. **描边外扩法线**：只用于外轮廓几何膨胀，通常需要比着色法线更平滑。把它直接替换为着色法线会破坏硬表面的材质表现。

反面外扩沿原始法线时，UV 缝、硬边和拆分顶点会朝不同方向移动而产生裂缝。社区 URP 实现会按同位置顶点平均法线，再转到切线空间压入顶点色 RG，[C2]；这是实用方法，但必须避免覆盖已有 AO、Ramp 偏移和描边宽度数据。

推荐顶点数据契约：

| 数据 | 建议用途 |
| --- | --- |
| `COLOR.rg` | 压缩的切线空间描边法线 XY，或独立 UV 中存储 |
| `COLOR.b` | 颈部/特殊不受光区域遮罩 |
| `COLOR.a` | 描边宽度权重 |
| `UV1.x` | 阴影阈值偏移或固有 AO |
| `UV1.y` | 材质/特效辅助数据 |

不要同时让不同工具“借用”同一通道。导入脚本必须声明通道版本，例如 `ToonVertexLayout_v1`。

### 4.2 控制贴图的推荐契约

下面是**项目自定义规范**，不是《原神》官方通道表：

| 纹理 | 色彩空间 | 建议通道 |
| --- | --- | --- |
| `BaseMap` | sRGB | RGB 固有色；A 仅在明确时用于透明/自发光/腮红之一 |
| `ControlMap` | Linear | R 高光强度/类型；G 固有 AO；B 高光形状/金属遮罩；A Ramp 行/材质 ID |
| `FaceMap` | Linear | R/A 面部阈值；B AO 或局部禁光；其余通道保留 |
| `RampMap` | 通常 Linear | U 为光照量，V 为材质行；分昼/夜或按环境混合 |
| `NormalMap` | Normal Map | 切线空间法线 |
| `MatCap` | 视内容决定，通常 sRGB 色彩 | 视空间法线查表的风格化高光 |
| `EmissionMask` | Linear | 自发光强度；颜色由 HDR 参数或独立贴图提供 |

Unity 官方说明：颜色纹理通常应启用 sRGB；法线、遮罩、LUT 等非颜色数据应关闭 sRGB，否则采样值会被自动线性化，阈值和材质 ID 会改变。[U6] Ramp 若存的是最终艺术颜色，可以按作者空间决定；若 U/V 或灰度被当作数值，必须关闭 sRGB。

不要一开始为 31 个材质分别绘制一套控制图。先按视觉响应归并为五类 Ramp 行：皮肤、浅色布料、深色布料/头发、金属饰件、眼睛/发光件。建议的第一版新增资产如下：

### 4.3 Ramp 的采样与导入设置

设归一化表面法线为 $N$、指向光源的方向为 $L$：

$$
d=N\cdot L,\qquad h=\frac{d+1}{2}.
$$

`h` 是半兰伯特量，只是把 $[-1,1]$ 映射到 $[0,1]$，不是更物理正确的漫反射。材质控制可写为：

$$
u=\operatorname{saturate}\left(\frac{h+a-t}{w}\right),\qquad
v=\frac{k+0.5}{K},\qquad C_r=\operatorname{Ramp}(u,v),
$$

其中 $a$ 是固有 AO/顶点偏移，$t$ 是明暗阈值，$w$ 是过渡宽度，$k$ 是材质行，$K$ 是总行数。采样行中心的 `0.5` 可避免双线性过滤混入相邻行。

Ramp 推荐设置：`Wrap Mode = Clamp`、关闭压缩或使用无损/高质量格式、关闭 Mip Map（角色始终清晰且行很窄时）或自制带行间 padding 的 mip、`Filter = Bilinear` 用于柔和过渡，`Point` 用于严格色阶。移动端压缩前要逐通道比较，否则 0.3/0.5/0.7 之类 ID 会被污染。

**[M2 项目实测修正]** 本项目的 Ramp 是与 BaseMap 相乘的线性颜色倍率，不是可以直接照搬的最终显示色。首版把亮端设为约 0.90–1.00，硬二分取消了 M1 连续 `NdotL` 的中间衰减，导致正面亮度几乎不降且侧面更亮；最终将五类亮端收敛到 0.82–0.93、暗端到 0.50–0.76。256×64 RGBA、五行、`V=(k+0.5)/5`、sRGB Off、Clamp、Bilinear、No Mip、Uncompressed 已由像素和导入器检查验证。该范围只对当前 BaseMap/固定曝光成立，不应外推为通用或官方数值。`toon_defo.bmp` 是 16×16 MMD toon 图，不满足本契约，故使用项目自绘 `Sandrone_Ramp_WarmCool.png`。

**[M2 实现差异]** 上式把阈值和宽度折入 Ramp 的 U；当前实现先用 `smoothstep` 得到明确 BandMask，再在 U=0.25/0.75 取暖/冷端点。这使 BandMask、边界 AA 和 Ramp 色本身可以分别调试。两种写法都是通用实现，当前选择不是对公开游戏 Shader 的逆向结论。

### 4.4 抗锯齿的二分边界

直接 `step(t, h)` 在运动、远景和斜边处会闪烁。可用屏幕导数创建约一个像素的软区：

```hlsl
float x = halfLambert + aoOffset;
float aa = max(fwidth(x) * _BandAA, 1e-4);
float lit = smoothstep(_Threshold - aa, _Threshold + aa, x);
```

这不是把阴影“做软”，而是对离散边界做解析近似抗锯齿。若使用 TAA，仍需避免阈值随纹理压缩噪声逐帧变化。

## 5. Unity URP 渲染架构

### 5.1 为什么选手写 ShaderLab/HLSL

Unity 6 官方将自定义 URP 光照分为 HLSL、Shader Graph Custom Function 和修改 URP 源码。Custom Function 不能增加 Pass、改变顶点输入结构或设置额外 Render State；面部/身体分支、ShadowCaster、DepthNormals、Stencil 和反面描边均需要这些能力，所以本项目应以 ShaderLab/HLSL 为主。[U1]

推荐的 Pass 结构：

```text
ShadowCaster          写入主光/附加光阴影图
DepthOnly             供深度、软粒子、后处理使用
DepthNormals          供屏幕描边/调试使用（启用时）
UniversalForward      身体、面部、头发主体着色
SRPDefaultUnlit       反面外扩描边
Hair/Eye Stencil Pass 仅在需要刘海透视或投影时启用
MotionVectors         使用 TAA/STP 且角色变形需要时实现
```

HoyoToon Built-in 版本实际包含 HairShadow、Character、额外光、Outline、ShadowCaster、Stencil 等多个 Pass，[C1]；这证明复杂角色渲染需要明确的 Pass 编排，但它的 `ForwardBase/ForwardAdd` 标签不能直接搬到 URP。

### 5.2 URP 主光、附加光和阴影接口

Unity 6 自定义 Shader 应包含：

```hlsl
#include "Packages/com.unity.render-pipelines.universal/ShaderLibrary/Lighting.hlsl"
```

使用 `GetMainLight(shadowCoord)` 获得 `direction`、`color`、`distanceAttenuation`、`shadowAttenuation`；`GetShadowCoord` 或 `TransformWorldToShadowCoord` 生成阴影坐标。阴影变体至少需要主光的 `_MAIN_LIGHT_SHADOWS`、级联/屏幕阴影关键字；物体还必须有 `ShadowCaster` Pass。[U2][U3]

**[项目实测修正]** 本工程 PC Renderer 是 Forward+。URP 17.5 的 `GetMainLight()` 在 Cluster Light Loop 分支把方向光 `distanceAttenuation` 设为 1；普通 Forward 分支读取逐物体 `unity_LightData.z`。Forward+ 不请求这份逐物体 LightData，因此自定义 Shader 若漏掉 `#pragma multi_compile _ _CLUSTER_LIGHT_LOOP`，方向/颜色仍可读但 attenuation 会错误为 0，BaseLit 成为黑色。M1 已用主光颜色/attenuation 诊断图复现并修正；该关键字是管线兼容变体，不是材质功能变体。升级 URP 时应重新核对本地 `Core.hlsl` 与 `RealtimeLights.hlsl`，不要把此结论无条件外推到其他版本。

```hlsl
// Varyings 仅在 URP 明确要求顶点阴影坐标时携带 shadowCoord。
#if defined(REQUIRES_VERTEX_SHADOW_COORD_INTERPOLATOR)
OUT.shadowCoord = GetShadowCoord(positionInputs);
#endif

// Fragment：级联阴影必须用当前像素的 positionWS 选择 cascade。
#if defined(REQUIRES_VERTEX_SHADOW_COORD_INTERPOLATOR)
float4 shadowCoord = IN.shadowCoord;
#elif defined(MAIN_LIGHT_CALCULATE_SHADOWS)
float4 shadowCoord = TransformWorldToShadowCoord(IN.positionWS);
#else
float4 shadowCoord = 0;
#endif
Light mainLight = GetMainLight(shadowCoord);
```

**技术决策：** 第一版只让主方向光决定 Toon 色阶；附加光用低强度加法或只影响高光/VFX。多个点光都参与硬阈值会在角色表面制造互相争夺的色块。Forward+ 的附加光循环与普通 Forward 不同，不能假设 `GetAdditionalLightsCount()` 在 Forward+ 返回实际数量。[U1]

### 5.3 间接光与风格一致性

完全忽略环境光会使背光面死黑；完全照搬 PBR GI 又会冲淡色阶。折中方案：

$$
C_{indirect}=C_{base}\,\bigl(k_a C_{ambient}+k_s\operatorname{SH}(N)\bigr)\,AO,
$$

再限制其最大亮度或仅用于暗部。URP 可用 `SampleSH` 采样光照探针，反射探针可用 `GlossyEnvironmentReflection`。[U7] 对动画角色，应至少放置 Light Probe；否则场景昼夜变化时角色与环境脱节。

## 6. 光照、阴影与 AO

### 6.1 需要分开的三种“阴影”

- **明暗分区（shading band）**：由 $N\cdot L$、Ramp 和阈值产生，描述表面朝向。
- **投射阴影（cast shadow）**：来自 Shadow Map，描述光被其他几何遮挡。
- **固有阴影/AO（painted or authored AO）**：艺术家在控制贴图或顶点中指定，如衣褶、头发根部、配件接触区。

推荐组合不是简单相乘全部压黑，而是先将投射阴影转换为稳定遮罩，再与风格阴影合并：

```hlsl
float formShadow = ToonBand(NdotL, control.g);
float cast = smoothstep(_CastShadowLow, _CastShadowHigh,
                        mainLight.shadowAttenuation);
float litMask = min(formShadow, cast);
float3 diffuse = lerp(baseColor * shadowTint, baseColor * lightTint, litMask);
```

如果 Shadow Map 的 PCF 软边直接乘到硬色阶上，会出现双重边缘。选择之一是：投射阴影保留柔边、表面明暗硬边；选择之二是把投射阴影也阈值化。前者更稳，后者更插画化。

**[M3 核验不一致，2026-07-16]** `SandroneShadowReceiverM3.shader` 已按上面的片元级分支修正，但 M3/M4/M5/M6/M8 角色 Forward Shader 仍无条件在顶点调用 `GetShadowCoord` 并插值后传给 `GetMainLight`。URP 17.5 只在 `_MAIN_LIGHT_SHADOWS`/screen 分支定义 `REQUIRES_VERTEX_SHADOW_COORD_INTERPOLATOR`；`_MAIN_LIGHT_SHADOWS_CASCADE` 的内建 Lit 路径在片元以 `TransformWorldToShadowCoord(positionWS)` 选择 cascade。故当前 PC 4 级 CSM 的角色接收阴影不能标为已验证，近景 Raw/Styled/Form/Final 截图也不能证明 cascade transition 正确。投射遮罩的 `smoothstep`/强度映射与 `min(formBand, castShadow)` 仍是已观察到的局部行为，但前提是输入 `shadowAttenuation` 正确。该修复跨 M3–M8，按重大问题等待统一方案确认。

**[M3 测试方法修正，2026-07-15 更新]** 阴影强度 0.85 时，线性 full-shadow attenuation=0.15，写入 sRGB PNG 约为 108 灰阶。因此“灰阶像素占比”不等于“软边占比”；验证器必须分离稳定遮挡平台、稳定受光区和两者之间的过渡。顶视正交相机下的平面 caster/receiver 会被 per-cascade caster culling 分到不同裁剪体并产生多边形空洞；该测试布局已废止。Alpha Clip 契约改用正面相机、竖直 caster/receiver 与相近相机深度，全程生产 4 级联，禁止临时切单级联制造通过。

### 6.2 主机公开方案与课程实现的差距

米哈游公开演讲说明主机版本为兼顾近景细节和大范围覆盖，最多使用 8 个阴影级联；前 4 级每帧更新、后 4 级交错更新；屏幕空间软阴影只在半影区域进行多次 Poisson 采样以降成本。[O1] 这是大型开放世界方案，不应作为课程角色 Demo 的起点。

课程实现建议：1 个方向光、2 或 4 级 CSM、合理阴影距离、正常 Bias/Normal Bias；用 Frame Debugger 确认 ShadowCaster 后，再增加角色胶囊 AO 或接触阴影。阴影痤疮增大 Bias，悬浮减小 Bias；二者不能同时靠一个极端数值解决，应同时调整阴影分辨率、级联范围和法线偏移。

### 6.3 日夜与天气

**[已证实]** 官方只确认实时昼夜/天气和按材质调整光照，[O1] 未公开角色具体 Ramp 切换规则。**[通用方案]** 为角色建立全局 `EnvironmentStyle`：

```text
sunDirection, sunColor, ambientColor,
dayNightBlend, weatherBlend,
warmRamp, coolRamp, shadowTint,
specularScale, rimColor
```

由场景系统一次更新，所有角色读取同一 Constant Buffer。昼夜应在 Ramp/暗部色/环境光之间连续插值，避免宏切换或瞬时换图造成跳变。

## 7. 身体 Toon Shading 与材质响应

### 7.1 漫反射主体

基础输出可写为：

$$
C_d=C_{base}\left[(1-m)C_{shadow}+mC_{light}\right]C_{lightColor}+C_{indirect},
$$

其中 $m$ 来自 Ramp 或离散色阶。Ramp 的价值是把“亮度函数”变成“艺术颜色查表”；X-Toon 论文进一步说明，第二维可编码深度、法线或细节等级，从而控制视角相关效果和抽象程度。[P3]

社区复刻对公开资源的常见解释是：控制贴图 G 作为 AO，A 选择 Ramp 行，顶点色偏移阴影阈值，[C3][C4]；应把它看作一个可用的**数据设计范式**，而不是不变的官方格式。

### 7.2 材质 ID 与高光策略

同一个角色中至少区分皮肤、头发、布料、丝绸、金属、玻璃/宝石。不要用单一 Blinn-Phong 处理全部区域。

| 材质 | 推荐响应 | 主要参数 |
| --- | --- | --- |
| 皮肤 | 极弱、宽而柔的高光；受遮罩控制 | `specIntensity`, `roughness` |
| 普通布料 | 关闭或很弱高光 | `specMask` |
| 丝绸/丝袜 | 宽高光 + Fresnel，暗部可保留少量 | `NdotV`, `specWidth` |
| 金属 | MatCap 或量化 Blinn-Phong，颜色随 Base | `matcap`, `metalMask` |
| 宝石 | HDR 高光 + 自发光 + Bloom | `emission`, `fresnel` |

米哈游 2018 年《崩坏3》演讲公开展示了漫反射/高光多通道 2D Ramp、丝绸多层各向异性高光、材质专用反射与轮廓色。[O2] 它为“按材质族拆光照模型”的工程方法提供一手依据；本文只借鉴方法，不据此宣称【桑多涅】或《原神》使用相同通道。

Blinn-Phong：

$$
H=\frac{L+V}{\lVert L+V\rVert},\qquad S=\max(N\cdot H,0)^p.
$$

风格化时对 $S$ 再做 `smoothstep` 或 Ramp 映射。社区实现还用视空间法线 XY 采 MatCap，叠加到遮罩限定的金属区域。[C2] MatCap 便宜且可控，但高光锁定相机而非真实光源；角色大幅转动、镜头上下移动时要检查“贴在屏幕上”的感觉。

### 7.3 颜色设计而非只调亮度

Gooch 等人的 NPR 光照模型强调用冷暖色相变化表达朝向，并为轮廓和高光保留最暗/最亮动态范围。[P1] 对本项目的启示是：暗部不应只乘黑色。优先为暗部设置略冷或环境相关的色相，同时保护 BaseMap 的局部色彩关系。

## 8. 面部、眼睛与头发

### 8.1 面部为何不能直接使用网格法线

低模鼻、嘴和眼眶的逐像素 Lambert 阴影会产生写实但不美观的小碎影。面部需要稳定、可设计、随主光方向连续变化的二维阴影形状。通用方案有三类：

1. 编辑脸部顶点法线，使其近似球面或统一前向；简单但局部形状控制弱。
2. 用代理球/椭球法线场计算光照；连续、无需贴图，但鼻翼等艺术阴影不易控制。
3. 用 Face SDF/阈值图；艺术可控，需制作数据和稳定的头部坐标。

社区复刻和 HoyoToon 均实现第 3 类：[C1][C2][C3]。因此它是本项目推荐方案，但“《原神》所有版本都以同样方式实现”仍属于未证实推广。

### 8.2 Face SDF 的稳健算法

设头部世界空间前向 $F$、右向 $R$，把光方向投影到头部水平面：

$$
L_h=\operatorname{normalize}\left(L-(L\cdot U)U\right),
$$

$$
f=L_h\cdot F,\qquad r=L_h\cdot R,
\qquad q=\frac{1-f}{2}.
$$

`r` 决定是否水平翻转 UV，`q` 是阈值。令贴图值为 $s(u,v)$：

```hlsl
float2 faceUV = (r >= 0) ? uv : float2(1.0 - uv.x, uv.y);
float sdf = SAMPLE_TEXTURE2D(_FaceMap, sampler_FaceMap, faceUV).r;
float aa = max(fwidth(sdf) * _FaceAA, 1e-4);
float faceLit = smoothstep(q - _FaceSoftness - aa,
                           q + _FaceSoftness + aa, sdf);
```

Valve 的距离场工作证明，低分辨率距离场配合阈值可在放大时生成平滑锐利边缘，并扩展到描边、阴影等效果。[P4] Face SDF 与字体 SDF 的语义不同：它编码的通常是“光角变化对应的面部遮罩阈值”，不是三维脸表面的真实距离。

制作流程：在 Blender 中固定相机和头部，对若干水平光角手工绘制/渲染理想阴影遮罩；对每张遮罩求二维有符号距离；按角度合成单张阈值图；在 Unity 中用头部局部光角扫一遍验证单调性、左右翻转和鼻翼区域。FaceMap 是数据图，关闭 sRGB，通常关闭压缩并使用 Clamp。

【桑多涅】应从骨骼 `頭` 建立头部坐标，不从根骨或世界轴取方向；`颜.png` 只是手绘 BaseMap，不能直接作为 Face SDF。第一版可先使用“编辑脸部法线 + 几乎不受 Lambert 影响”的方案完成 M2，再在 M5 制作独立 `Sandrone_Face_SDF.png`。验收必须覆盖正面、左右 90° 和背光，并检查脸与 `首/肌` 的暗部颜色是否连续。

### 8.3 眼睛、睫毛和刘海

建议渲染顺序：眼白/虹膜不透明或 Alpha Clip -> 睫毛 Alpha Clip -> 前发不透明/Clip -> 可选刘海半透明叠加。Alpha Clip 比 Alpha Blend 更稳定地写深度和投影；真正半透明仅用于需要看穿刘海的局部。

“透过刘海仍能看见眼睛”有三种实现：

- 眼睛/脸写 Stencil，刘海在指定区域降低 Alpha；可控但需要严格模板值和 Pass 顺序。
- 刘海根据视角与眼睛遮罩渐隐；实现简单，侧视时要抑制。
- 独立眼睛覆盖 Pass；最稳定但容易穿过不应穿透的配件。

HoyoToon 当前 Shader 提供 Face/Eye/Hair stencil 分支和刘海混合参数，[C1]，可作为状态编排参考。透明排序问题要在多个角色重叠、侧脸、镜头穿越时测试，不能只看单角色正面。

标准 `桑多涅.pmx` 已把眼睛拆成 `白目/目/目2/目3/目al+/目光/星目` 多层；`目al+` 通过材质 Morph 从 Alpha 0 启用。导入 Unity 后若只看到基础瞳孔，先检查材质 Morph 是否丢失，而不是立刻调整深度。建议顺序为：`白目` 写深度 → `目/目2/目3` Opaque 或 Clip → `目al+/目光/星目` 局部叠加；每层使用独立 Object/Stencil ID，避免穿过眼睑和刘海。

内层“目齿轮可动版”将眼齿轮拆成额外材质/几何，并在 PMX 注释中声明远距离锯齿。采用它时必须做 0.5 m、2 m、5 m、10 m 距离测试；为细小 Alpha 图启用合适 Mip Map/Alpha Coverage，比较 SMAA/TAA，并提供退回标准眼部或切换 LOD 的路径。不要为了演示可动齿轮替换标准主体的全部 M0–M5 资产。

### 8.4 头发高光与刘海投影

头发天使轮首先是图形设计，其次才是物理反射。两条实现路线：

- **遮罩驱动**：高光形状纹理或 UV 条带乘量化 `NdotH`/`NdotV`，稳定、可画，最适合大块发片。
- **各向异性**：用发丝切线 $T$ 构造沿纤维方向的高光。Kajiya-Kay 是经典近似；Marschner 模型可表达多重高光和纤维轴旋转变化，但对本项目通常成本和复杂度过高。[P5]

推荐先用遮罩驱动，再用一个低成本各向异性项让高光随光源轻微移动。高光需要亮/暗区遮罩：完全不衰减会像自发光，完全乘硬阴影会让天使轮被切断。

**【桑多涅】决策：** `髮.png` 已绘入大块明暗，`髮+` 又以 `sp.png + hair_s.bmp` 提供 MMD 叠层。M6 首先在固定灯光下复现该叠层的形状，再将其简化为“美术遮罩 × 低频切线高光”；不要直接叠加完整各向异性模型，否则会与 BaseMap 既有高光形成双重亮带。米哈游 2018 演讲中的切线各向异性、双频高光和条带抖动可作为后续升级参考，[O2] 但不是本模型已有数据。

刘海投影可使用：

1. 真正 Shadow Map：几何正确，但脸上容易出现锯齿和碎影。
2. 屏幕/光方向偏移的刘海遮罩：把刘海轮廓投到脸上，稳定且可调。
3. Stencil 专用 HairShadow Pass：便于只影响脸，不污染背景。

项目优先 2 或 3；将偏移量按屏幕分辨率、FOV 和脸部尺度归一化，避免远近变化。

## 9. 描边与边缘光

### 9.1 三类描边方案的取舍

| 方案 | 优点 | 局限 | 适用 |
| --- | --- | --- | --- |
| 反面外扩（Inverted Hull） | 颜色/宽度可按材质和顶点控制；角色轮廓稳定 | 多一次几何 Pass；凹面、硬边、薄片会出错 | 角色主外轮廓 |
| 深度/法线边缘检测 | 逐像素、可覆盖场景和交界边 | 难区分材质颜色；噪声、透明和 MSAA 边缘复杂 | 场景轮廓、角色接触线 |
| 贴图内描线 | 美术完全控制，成本低 | 不随真实遮挡产生新线 | 膝盖、衣褶、五官线 |

Saito 与 Takahashi 早期工作已经提出把深度、法线等几何属性存入 G-buffer，再用二维后处理增强不连续和轮廓。[P2] 社区角色方案通常选择反面外扩，并用屏幕方法补接触线。[C2][C3]

### 9.2 反面外扩的正确空间

基础做法：额外 Pass `Cull Front`，沿描边法线外扩后输出深色。ShaderLab 的 `Cull Front` 会剔除朝向相机的正面，显示膨胀模型的背面壳。[U8]

世界/模型空间固定宽度会随距离变化。若目标是近似恒定像素宽度，应在裁剪空间偏移 NDC：

```hlsl
float4 posCS = TransformObjectToHClip(positionOS);
float3 nVS = TransformWorldToViewDir(normalWS);
float2 dir = normalize(nVS.xy + 1e-6);
float2 pixelToNDC = 2.0 / _ScaledScreenParams.xy;
posCS.xy += dir * (_OutlinePixels * widthMask) * pixelToNDC * posCS.w;
```

透视除法后偏移约为像素常量；正交相机则不依赖 `posCS.w`，需单独分支。还应限制极端掠射角和近裁面，避免尖刺。

关键参数：

- `OutlinePixels`：目标像素宽度，而不是世界单位；
- `widthMask`：顶点色 A、独立贴图或材质常量；
- `outlineNormal`：平滑描边法线；
- `depthOffset`：谨慎使用，过大会让线穿过前景；
- `outlineColor`：从 BaseMap 暗化或按材质指定，不宜全角色纯黑。

米哈游 2018 演讲明确把连续顶点法线、顶点色宽度控制和透视校正线宽列为几何描边要点，并用深度/法线检测的图像空间描边处理场景。[O2] 这与本文“角色 inverted hull + 场景/接触线屏幕检测”的组合一致，但仍应在【桑多涅】的裙摆薄片、发梢和透明叠层上逐材质禁用或限幅。

**效果对比：** 纯黑等宽线有漫画印刷感，但会压扁浅色角色；按 Base 色相压暗的彩色描边更接近插画。脸部和亮色头发通常使用更浅、更细的线，鞋底和深色配件可更重。

### 9.3 屏幕空间边缘检测

对线性眼空间深度 $z$ 和法线 $N$ 做邻域差分：

$$
e_z=\max_i|z_i-z_0|/\max(z_0,\epsilon),\qquad
e_n=\max_i(1-N_i\cdot N_0),
$$

$$
e=\operatorname{saturate}(w_z e_z+w_n e_n).
$$

深度阈值应随距离归一化；法线阈值应防止平滑曲面产生满屏线。透明对象通常不在 DepthNormals 中，需要另画 ID/Mask 或排除。Unity 的深度纹理由拥有 ShadowCaster/DepthOnly 等有效 Pass 的对象产生；缺 Pass 的对象会在深度中消失。[U9]

### 9.4 边缘光不是描边

Fresnel 型边缘光：

$$
r=(1-\max(N\cdot V,0))^p.
$$

它描述掠射角亮度，随表面法线变化；描边是几何或图像边界。社区方案还会沿屏幕法线偏移采样深度，以固定像素宽度判断边缘光区域。[C2] 该方法对背景深度敏感，应只在角色 Layer 上启用，并对远距离衰减。

## 10. 特效（VFX）

角色 NPR 特效应服从同一色彩和形状语言：大色块、可读轮廓、少量高亮中心、受控噪声，而不是简单叠加写实粒子。

### 10.1 常用模块

- **自发光**：`EmissionMask * HDRColor * intensity`。先在材质输出 HDR 值，再由 Bloom 扩散；不要把模糊光晕直接画进 BaseMap。
- **溶解**：噪声 $n$ 与阈值 $t$ 比较，`clip(n-t)`；边缘带为 `smoothstep(t, t+w, n)-step(t+w,n)`，用于 HDR 边光。阴影 Pass 必须应用相同 clip，否则实体消失后阴影仍在。
- **Fresnel 护盾/能量壳**：Fresnel 乘深度交界高亮和流动噪声。透明壳通常 `ZWrite Off`，要限制重叠层数。
- **UV 流动与 Flipbook**：用于符文、火焰和冲击波。帧间插值可减跳变，但增加采样。
- **软粒子**：比较粒子深度与场景线性深度，靠近交界时淡出，避免方片穿插。
- **屏幕扰动**：在透明 Pass 中偏移 Scene Color UV；对 UI、角色线条和镜头边缘设置遮罩，避免整屏发糊。
- **Trail/Ribbon**：剑光与运动线优先使用带状几何，沿寿命控制宽度、颜色、透明度和 UV。

【桑多涅】的低风险 VFX 顺序是：先用 `目光` 做受遮罩眼部 Emission，再用 `大剑.pmx/Mat_Cyrstal` 验证 Fresnel + Bloom，最后用 `显示屏.pmx` 的 `显示屏2.png` 验证透明发光层。三者都通过独立 `EmissionMask` 进入 Bloom；`脸红.png`、白衣和皮肤高光禁止进入。这样可在不引入粒子系统的情况下完成“材质 HDR → Bloom → 色调映射”的最小闭环。

### 10.2 渲染顺序与状态

建议顺序：不透明角色 -> Alpha Clip 头发/睫毛 -> 描边 -> 局部透明角色材质 -> 不透明场景后的 VFX -> 全屏后处理 -> UI。透明 VFX 默认不写深度；需要自身遮挡时可增加只写深度的预 Pass，但非凸粒子会出现排序副作用。

Unity 6 的 Visual Effect Graph 在官方文档中仍说明对 URP/移动端的完整支持在开发中。[U13] 面向课程和跨平台项目，优先使用 Particle System + Shader Graph/HLSL；需要大量 GPU 粒子且目标平台明确时再评估 VFX Graph。

## 11. 后处理与最终合成

### 11.1 推荐顺序

```text
HDR 不透明/透明合成
  -> 可选角色/场景屏幕描边
  -> Bloom 提取、降采样、模糊、升采样
  -> Exposure / White Balance / Color Grading
  -> Tonemapping
  -> 抖动/颗粒/半调/像素化等显示风格
  -> TAA/FXAA 或按具体算法调整顺序
  -> UI
```

抗锯齿的具体位置取决于引擎实现。TAA 需要 Motion Vector 并可能模糊细线；FXAA 成本低但也会软化 Toon 边界；MSAA 主要解决几何边缘，不能解决 Shader 内 `step` 的锯齿。调试时逐项关闭，不要用后处理掩盖材质错误。

### 11.2 Bloom

标准 Bloom 三步：提取亮区 -> 多尺度模糊 -> 加回原图。社区 URP 复刻仓库也采用阈值提取、分级降/升采样和最终混合，[C5]；Unity 官方 Bloom 提供 `Threshold`、`Intensity`、`Scatter`、`Tint`、`Clamp`。[U10]

技术决策：

- 角色眼睛和宝石用独立 Emission Mask，避免皮肤高光也进入 Bloom；
- `Threshold` 只允许 HDR 强光进入，BaseMap 常规白色不应泛光；
- `Clamp` 限制爆炸特效污染整幅画面；
- 在不同曝光、昼夜和分辨率下检查 Bloom 半径；卷积核应与分辨率归一化。

### 11.3 色调映射和色彩分级

**[已证实]** 米哈游公开分享称，其主机 HDR 管线没有直接采用 ACES RRT+ODT，因为不符合目标风格；团队还处理了资产依赖 tone mapping 色相偏移的问题，并把相关计算合入 LUT。[O1] 这说明“行业标准”不能替代风格验证。

项目流程：

1. 全部材质和光照在 Linear 空间计算；HDR 输出也要求 Linear 色彩空间。[U12]
2. 先锁曝光，再调材质；曝光变化会改变 Bloom 和高光判断。
3. 在 Neutral、ACES 或自定义曲线之间做 A/B 对比，重点观察皮肤色、红色服装、蓝紫暗部和高饱和特效。
4. LUT 只做全局协调，不修复单个材质错误。
5. SDR 截图和目标显示器实机都要验收；不要只看 Editor Game View。

### 11.4 课程要求的风格化后处理

最低风险选择是 Bloom；如果要体现“可演示模块”，可增加以下之一：

- **像素化**：先低分辨率渲染，再最近邻放大；UI 独立渲染。可加像素对齐相机避免移动闪烁。
- **半调/仿色**：根据线性亮度选择点阵或 Bayer 阈值；图案使用屏幕像素坐标，避免相机移动时爬动。
- **故障艺术**：对局部通道做时间量化的 UV 偏移，并用噪声/对象 ID 限定区域。
- **素描/水彩**：需要法线、深度、纸张纹理和时域稳定性，开发成本明显高于课程最低要求。

## 12. Unity URP 可执行实现流程

### 12.1 项目基线

截至 2026-07-14，官方最新 LTS 为 Unity 6.3 LTS，支持至 2027 年 12 月；Unity 6.0 LTS 将于 2026 年 10 月结束常规支持。[U17] 官方同时把 Unity 6 Update releases 定位为 production-ready；本项目按用户明确要求锁定 `6000.5.3f1`，不再套用旧版“建议 6000.3.x”的默认选择。实际内置 URP 包为 `17.5.0`，版本与报告都写入工程，后续 Phase 不得无记录升级。

实践修正：`6000.5.3f1` 随附的 cross-platform 模板包含旧 Timeline/Version Control 等可选包，首轮编译分别触发 `Object.GetInstanceID()` 与非泛型 `TreeView` 的 CS0619 error；因此 M0 的 `Packages/manifest.json` 只保留 `com.unity.render-pipelines.universal: 17.5.0`，由 Package Manager 解析其必要依赖。模板原始请求为 17.0.1，但 Package Manager 在该 Editor 中实际解析为内置 17.5.0，故以 lockfile 实值为准。结论仅适用于本次模板/版本组合，不推广为所有 Unity 模板都应删除可选包。

基础设置：

1. Unity 工程位于 `<PROJECT_ROOT>/Unity/SandroneToon/`；创建 URP 3D 项目，Color Space 设为 Linear。M0 标定相机关闭 HDR 以稳定验证 BaseMap；HDR 在 M8 Emission/Bloom 开始时再启用并单独验收。
2. 第一版使用 Forward 渲染路径、单主方向光；URP Asset 启用 Main Light Shadows。Depth Texture、Opaque Texture、Additional Lights 只在对应模块开始时启用，避免一开始无法判断额外成本来源。
3. 建立 `Character`、`CharacterFX`、`Environment` Layer 和专用 Volume Layer；为屏幕描边准备 Character Rendering Layer/Object ID，而不是靠颜色猜对象。
4. 创建 `ToonCalibration.unity`：中性灰背景、18% 灰球、固定曝光、固定焦距相机、角色旋转台和可绕头 360° 的方向光。所有 A/B 截图使用同一相机和 Volume。
5. 先导入由标准 `桑多涅.pmx` 生成的 FBX；`Normals = Import`、`Blend Shape Normals` 与基础法线策略一致、`Tangents = Calculate Mikktspace`、`Index Format = Auto`。[U4]
6. 检查根节点位置/旋转为零、角色正向和目标身高；MMD 单位不直接等于 Unity 米制，按项目标尺校准，不写死通用 Scale Factor。
7. BaseMap 开 sRGB；Control/Face/EmissionMask/Ramp 数值图关 sRGB；法线图设 `Normal map`。Unity 官方要求 lookup/mask 这类数值纹理绕过 sRGB 采样。[U6]
8. 建立 `SandroneMaterialMap.asset` 或等价配置，显式把 31 个 PMX 材质映射到 Face/Eye/Hair/Skin/Cloth/Metal/Overlay；禁止按贴图文件名猜类别。
9. 暂不合并材质和删骨。M9 已记录原始 31 槽、Pass/Stencil/透明状态和 692 骨骼基线；因状态不兼容且没有覆盖物理/表情/挂点的动作语料，当前拒绝合并和剥离。后续只能按兼容状态分组并建立动作回归后再实验。

### 12.2 按可验证里程碑实现

| 里程碑 | 主要输入 | 输出/实现 | 通过条件 |

| M4 材质响应 | `Sandrone_Body_Control`、材质族 | 皮肤/布料/金属/袜/发叠层 | 高光不泄漏到未遮罩区域；`饰`、`袜+`、`髮+` A/B 可单独关闭 |
| M5 面部 | `頭` 轴、`Sandrone_Face_SDF` | 面部阈值阴影、颈部颜色衔接 | 水平光角 360° 连续；左右翻转正确；正面/3/4 无鼻嘴碎影；`颜/首/肌` 暗部不脱节 |
| M6 眼睛/头发 | 眼部多层、前后发、Stencil/Clip | 眼叠层、刘海投影、遮罩高光 | 正/侧/俯视无错误穿透；`Eye AL` 和腮红可动画；可选齿轮变体在 0.5–10 m 完成抗锯齿评估 |
| M7 描边 | 平滑描边法线、宽度权重 | Inverted Hull；可选屏幕接触线 | 1080p 下 1/3/10 m 与不同 FOV 的目标宽度偏差可接受；发梢/裙摆无大面积尖刺或内线泄漏 |
| M8 VFX/Bloom | `目光`、晶体、显示屏 EmissionMask | HDR Emission、Bloom、可选粒子 | Bloom Extraction 中只有指定区域；关闭 Bloom 后 Base 色不改变；溶解/Clip 与 ShadowCaster 同步 |
| M9 合成/性能 | 全部 Pass、Volume、目标平台 | 色调映射、AA、变体剥离、性能报告 | 同相机 A/B 截图齐全；无粉色材质/透明排序回归；记录 CPU/GPU、Draw Call、Overdraw 与变体数；外存带宽若引擎不暴露则明确标为 PIX/RenderDoc 待测 |

顺序是依赖关系，不是功能清单。M0 未通过时不要调 Ramp；M1 头部轴未通过时不要制作 Face SDF；M6 透明/Stencil 未稳定时不要用 Bloom 掩盖眼部排序；M7 前不应把 31 材质激进合并。

### 12.4 材质校验器应自动检查

- 当前角色是否来自标准 `桑多涅.pmx`，眼齿轮变体是否被显式标记为可选；
- 31 个 PMX 材质是否全部映射，是否错误地把 `tex/sp.png` 当作 ILM/ControlMap；
- `目al+`、`照れ+` 等初始 Alpha 为 0 的层是否仍保留可动画控制；
- `頭` 骨与 Head Forward/Right/Up 是否存在且互相正交；
- Control/Face/Ramp 是否误开 sRGB；
- Ramp 是否 Clamp、压缩是否关闭、尺寸/行数是否匹配；
- Face 材质是否绑定头部轴；
- Alpha Clip 材质的 ShadowCaster 是否使用同一 Cutoff；
- 描边是否有宽度数据和平滑法线；
- 材质 ID 是否超出 Ramp 行数；
- Emission 是否为 HDR 且有 Mask；
- Shader Keyword 是否使用本地关键字并存在无效组合。

## 13. Blender 预览与资产制作流程

Blender 的快速 LookDev 可使用 Eevee：`Diffuse BSDF -> Shader to RGB -> Color Ramp -> BaseColor Multiply`。官方文档明确 `Shader to RGB` 仅 Eevee 可用，适合 NPR，但会破坏 PBR 并使屏幕空间反射、SSS、Alpha Clip 等上游效果不可预测。[B2] 因此它用于预览色阶和绘制贴图，不是 Unity 最终效果的逐像素参考。

执行步骤：

1. 固定一套与 Unity 相同方向的 Sun、环境色和相机焦距。
2. 为身体建立两级或三极 Color Ramp，先确认大色块，再绘制 ControlMap。
3. 为脸创建代理材质：用头部局部光角驱动 Map Range/Color Ramp，或在 Geometry Nodes/Python 辅助下预览 Face SDF。
4. 用 Data Transfer/Normal Edit/Weighted Normal 等工具处理艺术法线；导出前复制一份原始网格。
5. 将描边法线烘到明确的 Color/UV 属性；导出后在 Unity 调试视图验证数值是否保留。
6. 使用 Freestyle/反面壳仅作轮廓预览；最终宽度仍以 Unity 像素单位验收。
7. FBX 导出保留自定义法线、顶点色、骨骼和需要的 Shape Key；禁用不需要的 All Actions 和 Leaf Bones，避免文件膨胀和多余骨。[B1]

### 14.3 性能与变体

主要成本不是某个 `dot`，而是额外 Pass、纹理带宽、透明 Overdraw、全屏效果和 Shader Variant。

- 标准【桑多涅】有 31 个材质槽，单次 Forward 就至少形成相应的子网格绘制；若所有槽都参与 Forward + Outline + ShadowCaster，理论上可接近 93 次角色绘制事件，尚未计 DepthNormals/Stencil。先以 Frame Debugger 实测，再合并 Pass 状态完全相容的身体材质。
- 692 根骨骼和大量发/配件次级骨会增加动画、蒙皮与 Transform 成本。课程静态展示可烘焙姿态或剥离未使用骨；需要动画时按动作集做骨骼使用审计。
- 多张 2048² Base/叠层纹理在未压缩 RGBA32 下单张约 16 MiB（不含 mip）；必须按目标平台设置压缩和 Max Size。但 Control/Face/Ramp 这类阈值数据的压缩误差优先级高于节省的容量，应单独制定格式。
- 反面描边增加一次角色几何提交；屏幕描边增加全屏采样。混合方案应设置质量级别。
- Face/Body/Hair 不要组合成所有特性全开的单一“巨型 Shader”而不做剥离。
- `shader_feature_local` 用于材质可裁剪特性；运行时必须动态切换的全局状态不能依赖可能被剥离的变体。
- Unity 官方指出变体数量是各关键词集合大小的乘积；10 个二值集合可产生 1024 个组合。剥离可减少构建时间、文件大小、加载时间和内存。[U16]
- 移动端优先减少采样、透明层和全屏 Pass；桌面端再启用屏幕边缘光、多尺度 Bloom 和高质量阴影。

## 15. 技术决策清单与推荐基线

### 15.1 推荐的课程/原型基线

- 本项目固定 Unity `6000.5.3f1` + 随附 URP `17.5.0`，Linear 色彩空间；不为追随当前 LTS 标签迁移 Editor。新建的独立课程可另选固定的 Unity 6.3 LTS 补丁版，但不能与本项目的验证结果混称；
- 以外层标准 `桑多涅.pmx` 为角色基线，人偶、凳子、大剑和显示屏按独立 FBX 导入；眼齿轮可动版只作可选升级；
- 保留 31 个材质分区、眼睛/脸红材质变形和 `Clockwork Rotation` 骨骼变形，跑通动作后再裁骨或合批；
- 单主方向光，角色 Toon 只受主光控制；
- 复用原 BaseMap，另建 `Sandrone_Body_Control`、`Sandrone_Face_SDF`、暖/冷 Ramp、Emission Mask 与描边宽度顶点色；`sp.png` 只保留为原叠层纹理；
- 身体二分/窄过渡 Ramp，投射阴影保留适度柔边；
- 面部先以编辑法线建立 M2 基线，再升级 Face SDF；眼睛/睫毛 Alpha Clip，并核对 `目al+`、`照れ+` 的变形驱动；
- 头发先复现 `髮.png + 髮+ / sp.png` 的原叠层关系，再加低频遮罩高光，暂不实现完整物理各向异性；
- 反面外扩彩色描边，宽度以像素控制；
- 眼部发光、大剑晶体和显示屏使用 HDR Emission + URP Bloom，皮肤、白布和脸红不得进入 Emission Mask；
- Neutral 或自定义轻量色调映射；
- 完整调试视图、Frame Debugger 验证和目标机性能记录。

该基线直接满足作业最低要求，并为后续面部、材质、VFX 和后处理升级保留接口。

### 15.2 何时升级

- 只有在 Ramp、Shadow、Face、Outline 都稳定后，再加入多光源和复杂 GI。
- 只有当屏幕尺度下确实看得到差异时，再给布料和皮肤增加法线细节。
- 只有当头发高光需要随动画和光源精确移动时，再升级到切线各向异性。
- 只有当场景也需要统一线稿时，再加入 DepthNormals 全屏描边。
- 只有当目标平台和效果规模证明 Particle System 不够时，再采用 VFX Graph。

## 18. M5 项目实践修正（Unity 6000.5.3f1 / URP 17.5.0）

- **已验证**：标准资产没有 Face SDF；`T_Face.png` 是 sRGB BaseMap，不能改作阈值图。M5 新增 `Sandrone_Face_SDF.png` 是项目自有、可替换的制作种子，生成器只在文件缺失时写入。
- **算法边界**：FaceMap R 编码“水平光角的艺术阈值场”，不是脸表面的三维欧氏距离。Shader 使用 `頭` 骨的 Forward/Right/Up，把主光投影到头部水平面并计算 `q=(1-f)/2`。原先按 Right 正负硬切 `uv/1-uv.x` 会在 `dot(Lh,Right)=0` 整脸跳变，现改为双采样并在宽度 `0.10` 的符号邻域连续混合；接近垂直光时回退到 HeadForward。
- **组合修正**：Face SDF 只替换脸部 form-light mask，最终仍使用 `min(face/form mask, styled cast shadow)`。因此启用 M5 不会关闭刘海/角色自阴影或地面投影，也不改变 ShadowCaster、Bias、Cascade 和接收面。
- **材质作用域**：仅 PMX 槽 0/1（`颜/颜2`）绑定 M5 Shader 并设置 `_UseFaceSDF=1`；槽 2–30 直接引用对应 M4 Material 资产。`首/肌` 槽 17/18 因而原样保留 M4 Skin 响应，用作脸颈暗部连续性验收，避免身体皮肤误采 FaceMap。
- **抗锯齿与导入**：FaceMap 为 Linear、Clamp、Bilinear、无 Mip、无压缩；阈值使用最小 `_FaceSoftness=0.02` 加 `fwidth(sdf)*_FaceAA`。最终手绘资产完成后应重测远距闪烁，再决定是否启用 Mip，而不是预先把压缩噪声带入阈值。
- **验证方法修正**：角色脸部只占全帧很小面积，整帧 MAE 会把真实响应稀释。M5 使用同机位侧向光和槽 0/1 Alpha-aware mask，保持 target MAE `>1`、non-target `<0.5`；最终为 `3.112/0`。左右光、头骨旋转和合成 FaceMap 替换也必须产生独立 Hash/MAE。
- **Pass/CBUFFER 修正**：`UsePass` 不只要求属性名存在；所有使用 `UnityPerMaterial` 的 Pass 必须共享兼容布局。M5 曾删除/重排 M4 字段，重新编译后裙摆槽 MAE 最高 13.670。现保留完整、同顺序 M4 前缀，M5 字段只追加；逐槽回归与近黑连通域成为硬门。
- **当前证据**：M5 Validator `94/94`；Special Audit 非脸/裙摆 MAE `0/0`，新黑像素/连通块 `0/0`，最大边界步进 `0.658 px`，镜像过零/环绕 MAE `0.798/0.416`，Debug 17/17 语义通过。真实 Game View M4→M5 低半帧 MAE `0`、红裙像素 `93,464/93,464`；Frame Debugger 为 2 M5 Face Forward + 29 M4 non-face Forward + 69 ShadowCaster + 1 receiver，0 failure。GPU 时间与 SRP Batcher 仍待 Profiler。
- **长裙片最终收口**：CBUFFER 错位是已复现的直接故障源；让全部 31 槽经过 M5 Shader又扩大了影响面。最终除保留脸 Shader 的完整 M4 CBUFFER 前缀外，还将非脸 29 槽改为直接复用 M4 Material 对象。该最小结构性修复不修改模型、UV、骨骼、贴图、环境光或截图参数，并由 `ExactNonFaceM4Reuse` 与场景材质执行门防止回归。
- **错误贴图假设复核**：标准 PMX 的裙槽纹理索引、Unity 31 槽材质/BaseMap GUID、源/导入文件 SHA-256、ST、BaseColor、Control/Ramp/MatCap 及 MPB 覆盖均已逐槽审计。31/31 源哈希一致，绑定属性 MPB 覆盖为 0；真实 Frame Debugger 的 31 个 Forward 有效签名多重集与 Renderer 对齐，故排除“红贴图被错误黑贴图/材质替换”。
- **长裙片 Cull 根因复核**：逐槽隔离显示 slot 21 为黑色长裙层、slot 26 为轮廓重合的红色长裙层；Cull A/B 证明 slot 21 的大面积黑色是背面片元，slot 26 的红色是正面片元。slot 21 使用 Cull Off 时两层 Opaque 深度竞争并随视口栅格化翻转；仅将 slot 21 修正为 Cull Back 后，真实 Frame Debugger 记录 `ZWrite On / ZTest LessEqual / Blend One,Zero`，同条件 M4/M5 均稳定为红色。
